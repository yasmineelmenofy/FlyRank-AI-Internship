# Task API

## 1. The Problem

The goal was to build a REST API for managing tasks as part of the FlyRank Backend Internship Week 2 assignment. The API had to support full CRUD operations, store data in memory instead of using a database, return the correct HTTP status codes, and include Swagger documentation.

Although those were the required features, I didn't want the project to look like a typical CRUD assignment where everything is written in one file. I wanted it to be organized in a way that would still make sense if the project became larger.

## 2. What I Did

I organized the project into separate controllers, routes, and a data module using an MVC-style structure. This wasn't required by the assignment, but I had used a similar structure in a previous project and found it much easier to organize and manage.

I implemented the CRUD endpoints, validated missing and invalid input, and returned appropriate HTTP status codes for successful and failed requests.

Normally, once I finish the required work, I submit the assignment and move on. This time I wanted to change that habit. Instead of stopping, I added task filtering by completion status and title search using query parameters because I wanted the project to go beyond the minimum requirements.

I also documented the API with Swagger. Before this project I had never used it, and honestly I wasn't excited about learning another tool. It ended up being the biggest new thing I learned during the assignment. Seeing the API documented and being able to test every endpoint from one place changed how I look at API documentation.

Throughout the project, I made sure I understood every line before adding it. If I couldn't explain what a piece of code was doing, I didn't want it in the project. I also chose to stay within the assignment's scope instead of adding MongoDB or TypeScript, even though I was tempted to. I wanted the project to solve the assignment well instead of becoming a different project.

## 3. What Came of It

The final project includes a documented REST API with full CRUD functionality, input validation, search, filtering, and Swagger documentation.

The biggest thing I learned wasn't another CRUD implementation. It was slowing down instead of submitting as soon as the required work was finished. Spending extra time on structure, documentation, and additional features made the project feel much more complete.

I also learned Swagger for the first time and now understand why backend projects use it.

More importantly, I wanted this project to show more than working endpoints. I wanted reviewers to see that I think about project structure, understand the code I write, and add my own improvements instead of only implementing the minimum requirements.
